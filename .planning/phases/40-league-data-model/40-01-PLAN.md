# Phase 40: League Data Model

## Goal
Implement the `League` (or Competition) data structure to allow grouping of games and granular analytics.

## Requirements
- **LEAG-DATA-01**: Create a `League` entity associated with a `Season`.
- **LEAG-DATA-02**: Update the `Event` entity to support association with a `League`.
- **LEAG-DATA-03**: Implement a migration to create a "Main League" for all existing seasons and link existing games.

## Proposed Data Schema

### LeagueEntity
- `id`: UUID (Primary Key)
- `seasonId`: UUID (FK to SeasonEntity)
- `name`: string (e.g. "Spring 2026 Session", "State Cup")
- `type`: string (enum: 'league', 'tournament', 'session', 'cup', 'other')
- `isActive`: boolean (default: true)
- `createdAt`: date
- `updatedAt`: date

### EventEntity (Updates)
- `leagueId`: UUID (FK to LeagueEntity, nullable)
- This column will be mandatory for games in the UI, but nullable at the DB level for practices/tryouts.

## Implementation Tasks

### 1. Database Migrations
- [ ] Create `leagues` table.
- [ ] Add `league_id` column to `events` table with FK.
- [ ] Data Migration: Create a default "Main League" for every existing season and link all existing events of type `'game'`.

### 2. Backend (NestJS)
- [ ] Generate `League` entity.
- [ ] Update `Event` entity with `league` relation.
- [ ] Update `SeasonsService` to auto-create a "Main League" when a new season is created.
- [ ] Implement `LeaguesService` and `LeaguesController`.

### 3. Frontend (Angular)
- [ ] Define `League` interface in shared models.
- [ ] Create `LeaguesService` in data-access library.

## Success Criteria
1. The `leagues` table exists and is linked to `seasons`.
2. Existing games are correctly linked to a default league per season.
3. New seasons automatically come with a default competition group.
